# 法律确认流程

本文件说明 Favewise 当前用于收集贡献者法律确认的实际流程。

这是一份面向维护方的贡献接收操作参考，并非面向最终用户的产品说明文档。

## 1. 基本原则

对于任何非简单、非琐碎的代码、设计、文档、测试、脚本或素材贡献，在项目维护方确认所需法律确认已经完成之前，不应合并进入项目。

贡献者通常至少需要阅读并接受以下文件：

- [CONTRIBUTING.zh-CN.md](./CONTRIBUTING.zh-CN.md)
- [CLA.zh-CN.md](./CLA.zh-CN.md)

对于部分贡献，维护方还可以要求进一步补充：

- [COPYRIGHT-ASSIGNMENT.zh-CN.md](./COPYRIGHT-ASSIGNMENT.zh-CN.md)
- 雇主或客户授权
- 身份确认
- 单独签署的 PDF 或电子签署流程

## 2. Pull Request 阶段的确认

每位提交 Pull Request 的贡献者，都应完成 PR 模板中的法律确认部分。

这一步的作用主要是：

- 让贡献者明确知悉项目规则
- 记录贡献者就其提交行为作出的声明
- 为后续如需单独签署提供基础记录

但 PR 模板本身并不能替代项目维护方另行要求的协议或签署文件。

## 3. 什么时候只要求 CLA

在以下情形下，仅要求贡献者接受 CLA 往往已经足够：

- 小型或中等规模的非核心改动
- 文档类贡献
- 测试、示例和低风险修复
- 维护方认为只需取得充分商业授权、无需直接取得所有权的贡献

## 4. 什么时候建议要求版权转让

对于以下类型的贡献，维护方应认真考虑要求签署版权转让协议：

- 核心产品逻辑
- 与变现直接相关的功能
- 自动分类工作流
- 专有集成能力
- 高价值重构
- 预计会在未来商业产品中持续占据核心位置的代码
- 来自长期或重要外部协作者的贡献

## 5. 建议保留的记录

对于每一项被接受的外部贡献，维护方至少应保留以下信息：

- GitHub 用户名
- 法定姓名（如已提供）
- 电子邮箱
- 对应的 Pull Request 链接
- 确认日期
- 已接受的法律文件种类
- 已签署的 PDF、电子签署回执或邮件确认
- 如适用，雇主或客户授权文件

## 6. 当前建议执行规则

在正式法律管理流程建立之前，建议维护方至少遵循以下标准：

- 未完成 PR 法律确认，不合并
- 对重要代码，未完成 CLA，不合并
- 对战略性或核心代码，如果维护方希望未来商业化或转让时权利链条足够干净，未完成版权转让，不合并

## 7. 实际可用的签署方式

维护方可以采用以下方式收集签署：

- 签字 PDF
- DocuSign 或类似电子签署工具
- 明确引用具体协议和具体贡献内容的邮件确认
- 在维护方认为风险足够低的情况下，以 Pull Request 评论方式完成确认

对于价值更高或风险更大的贡献，仍建议优先使用签字 PDF 或正式电子签署方式。

## 8. 面向未来公司转移的准备

如果本项目未来被转让、融资、并入新公司或由新的关联主体运营，维护方应妥善保留完整记录，以便清楚说明哪些贡献已完成授权或转让，哪些贡献者仍需补充后续文件。

## 9. 法律意见

本文件仅是运营和流程层面的模板，不替代合格律师提供的正式法律意见。
